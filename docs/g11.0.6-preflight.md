## §8. Step 7 -- Verifier.verify_bundle keyword pattern (observation; facade design)

The signature recorded in §4 sub-observations dictates the new
module-level `verify(manifest, args)` shape. The thin facade
must compose:

```python
def verify(manifest, args) -> VerificationResult:
    """Module-level procedural facade introduced at v0.11.8.

    Delegates to ``Verifier(...).verify_bundle(...)`` after
    dispatching to a language-specific facade per
    ``_LANGUAGE_DISPATCH``. ``args`` is the argparse namespace
    from gate11/cli.py (carries bundle_path, module_path,
    force_refresh, expected_identity, expected_issuer,
    allow_any_identity).
    """
```

**Dispatch shape (per F-XR-5: positional CasmVerificationError):**

```python
_LANGUAGE_DISPATCH: dict[str, "Callable[..., VerificationResult]"] = {
    "python": _verify_python,
    "rust": _verify_rust,
}

def verify(manifest, args) -> VerificationResult:
    language = manifest.language
    handler = _LANGUAGE_DISPATCH.get(language)
    if handler is None:
        raise CasmVerificationError(
            "CASM-V-001",
            f"v1.0 supports language in (python, rust); "
            f"got {language!r}. Go support ships in Phase "
            f"G11.2; ONNX in Phase G11.3.",
        )
    return handler(manifest, args)
```

**D24 discipline note:** the dispatch shape uses the
single-trailing-return idiom (raise on miss, return on hit) per
the v0.8.0 finding. Do NOT structure as `if handler: return
handler(...); raise CasmVerificationError(...)` -- D24 doesn't
model `raise` as terminal and the all-paths-return analysis
fires MARAD P1.

The handlers `_verify_python` / `_verify_rust` live in their
respective `python_verification.py` / `rust_verification.py`
facade modules and re-import the dispatch back into
verification.py for binding. The two-tier import is a known
acceptable circular-import pattern (verification.py imports the
handlers lazily inside `verify`, not at module top-level) -- the
v1.3 prompt's F-RV-7 absorption documents this idiom.
